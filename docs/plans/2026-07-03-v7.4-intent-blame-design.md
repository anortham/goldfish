# Goldfish 7.4.0 — Intent-Blame Recall + Brief Refresh Nudge (Design)

**Date:** 2026-07-03
**Status:** Approved
**Target release:** 7.4.0 (minor)

## Problem

Two evidence-backed gaps, both observed in this repository:

1. **Zombie briefs.** The v7.0.0 subtract-sprint brief shipped in April 2026 with
   every success criterion met, yet in July it still surfaces as the *active
   strategic direction* on every `recall()`. The 7.1.0 stale-brief detector keys
   staleness off the newest checkpoint referencing the brief — but checkpoints
   auto-inherit the active briefId, so a finished brief that keeps getting
   checkpointed under stays "fresh" forever. The detector structurally cannot
   fire for exactly the failure mode it was built for.

2. **Unqueryable intent data.** Checkpoints already capture `git.files` and
   `symbols` in frontmatter, but there is no way to ask "why did
   `src/workspace.ts` change?" or "what decisions touched `recoverWorkspace`?"
   Mid-task, that is the recall an agent actually needs — intent-blame scoped to
   what it is about to edit. BM25 body search only covers it by accident.

## Non-Goals

- No auto-archiving or any mutation of brief status on disk (non-destructive,
  same principle as 7.1.0).
- No parsing of brief success criteria to detect "done" (storage stays dumb).
- No new MCP tool (the 3-tool constraint holds) and no new brief action.
- No storage format change; no migration.

## Feature 1: Brief refresh nudge (zombie briefs)

### Behavior

In `resolveActiveBrief` (src/recall.ts):

- Abandoned check amended (Codex review finding): activity for the existing
  `STALE_BRIEF_DAYS` (7) check becomes `max(brief.created, brief.updated,
  latest referencing checkpoint)` — previously `brief.updated` was ignored, so
  updating an abandoned brief could not un-suppress it. With this change,
  "update the brief" reaffirms it out of both notice states. Body suppression
  and the `staleBrief` notice are otherwise unchanged. Regression test: old
  brief updated recently, no recent checkpoints → not suppressed.
- New check: if the brief is *not* abandoned-stale but `brief.updated` is older
  than `BRIEF_REFRESH_DAYS` (**14**), return the brief **plus** a new
  `briefRefresh` notice. This catches **zombie** briefs — checkpoints flowing,
  direction document frozen.
- The two notices are mutually exclusive: abandoned suppresses the body; zombie
  annotates it.

### Types (src/types.ts)

```typescript
export interface BriefRefreshNotice {
  id: string;
  title: string;
  updated: string;          // ISO 8601 UTC — brief.updated
  daysSinceUpdated: number;
}

// RecallResult gains:
briefRefresh?: BriefRefreshNotice | null;
```

### Display (src/handlers/recall.ts)

One line appended directly under the active-brief section:

```
⚠️ Active brief not updated in 78d — still the direction? Update it to reaffirm, or complete/archive it.
```

Reaffirming = any `brief save`/`update` on that brief (refreshes `updated`).
No new action required. Header suffix stays `+ active brief`.

### Scope notes

- Single-workspace recall only (cross-workspace recall does not surface briefs,
  matching the 7.1.0 precedent).
- `limit: 0` (brief-only) recall also gets the notice.

## Feature 2: `file:` + `symbol:` recall filters (intent-blame)

### Interface

Two new optional params on the existing `recall` tool:

```typescript
recall({ file: "workspace.ts" })          // why did this file change?
recall({ symbol: "recoverWorkspace" })    // what decisions touched this symbol?
```

- `file` matches against checkpoint `git.files`: case-insensitive path-suffix
  match with a `/`-boundary guard — `file: "recall.ts"` matches `src/recall.ts`
  and `src/handlers/recall.ts`; `file: "handlers/recall.ts"` matches only the
  latter. `all.ts` must NOT match `recall.ts` (boundary guard).
- Normalization applies to BOTH sides via one shared helper (Codex review
  finding): trim, replace `\` with `/`, strip leading `./` and leading/trailing
  `/`, lowercase — then compare exact or `endsWith('/' + query)`. A Windows
  style query like `src\recall.ts` behaves identically to `src/recall.ts`.
- Known coverage limit (documented, not fixed in code): `git.files` is captured
  from the working tree (staged + unstaged + untracked) at checkpoint time, so
  a checkpoint saved on a clean tree after committing carries no files and will
  not match `file:` queries. The checkpoint skill's "checkpoint BEFORE
  committing" guidance is the mitigation; deriving files from `git.commit` is
  deferred until real usage shows the gap matters.
- `symbol` matches against the `symbols` frontmatter list: case-insensitive
  exact match on the whole symbol name.
- Both are single strings (not arrays), trimmed; empty/whitespace input is
  ignored like other optional string filters.

### Semantics

- Both join `hasStructuredFilter()` and `applyStructuredFilters()` in
  src/recall.ts, so an active filter forces a full-corpus scan (older matches
  are never lost to the most-recent-N window) — identical to the 7.2.0
  `type`/`tags` filter behavior.
- Compose with each other and with `search`, `type`, `tags`, `briefId`, and
  date windows. Work in single- and cross-workspace (`workspace: "all"`) recall.
- Checkpoints with no `git.files` / no `symbols` simply never match the
  respective filter.

### Display

When a `file` or `symbol` filter is active, matched checkpoints keep their
`Files:` / `Symbols:` lines even in compact (non-`full`) mode, so the agent
sees why each hit matched. Implementation: `presentCheckpoint` retains
`git` (files only relevant part shown by the handler) and `symbols` when the
corresponding filter is set.

### Wire surface

- `RecallOptions` / `RecallInput` / `RecallArgs` gain `file?: string` and
  `symbol?: string`.
- Tool schema in src/tools.ts documents both params with one-line examples.
  The recall tool description must stay under the 2,000-char Claude Code cap
  (enforced by the existing test in tests/server.test.ts).

## Acceptance criteria

- [ ] A brief with fresh checkpoints but `updated` > 14 days old surfaces with
      body + refresh nudge; `updated` ≤ 14 days shows no nudge.
- [ ] An abandoned brief (no checkpoints > 7 days) still suppresses the body
      with the existing stale notice; the refresh nudge does not double-fire.
- [ ] Abandoned-staleness activity includes `brief.updated`: an old brief
      updated recently (no recent checkpoints) is NOT suppressed.
- [ ] File-filter normalization applies to query and stored paths alike
      (`src\recall.ts` query matches `src/recall.ts` stored path).
- [ ] `recall({ file })` and `recall({ symbol })` match per the rules above,
      scan the full corpus, and compose with existing filters in single- and
      cross-workspace recall.
- [ ] Filtered results retain `Files:`/`Symbols:` lines in compact mode.
- [ ] Recall tool description ≤ 2,000 chars; all five version surfaces read
      7.4.0; CHANGELOG has a 7.4.0 section.
- [ ] `bun test` and `bun run typecheck` pass.

## Architecture Quality

- **Affected modules:** src/recall.ts, src/types.ts, src/tools.ts,
  src/handlers/recall.ts, skills + .agents/skills mirror.
- **Caller-facing interface:** two optional recall params, one optional
  RecallResult field. Smaller than the behavior it unlocks.
- **Depth/locality:** follows the 7.2.0 filter and 7.1.0 staleness extension
  patterns exactly; changes stay inside recall's filter/present pipeline.
- **Test surface:** through `recall()` / `handleRecall()` — the same interface
  callers use.
- **Rejected shortcuts:** success-criteria parsing (intelligence in storage),
  a new `blame` tool (3-tool constraint), auto-archiving (destructive).
- **Architecture risk:** low
