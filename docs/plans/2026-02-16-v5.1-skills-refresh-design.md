# Goldfish v5.1 — Skills Refresh + Workspace Resolution

**Date:** 2026-02-16
**Status:** Approved

## Context

Goldfish v5.0.7 is stable with 260 tests passing. The core MCP tools (checkpoint, recall, plan) are solid. This design addresses gaps identified during a brainstorming session reviewing the full feature set against real-world usage patterns.

### User Context

- Works across multiple projects, bounces between them during the day
- Needs standup reports that look both backward (what was done) and forward (what's planned)
- Uses Azure DevOps with a central project for work items and sprint planning (but items are high-level, not worth integrating)
- Uses the superpowers plugin which writes implementation plans to `docs/plans/` in projects
- Self-describes as not terribly organized and forgetful — the tool needs to compensate

### Key Insight

Plans now live in two places:
1. **Goldfish plans** in `.memories/plans/` — created via the plan MCP tool, useful for non-Claude Code clients
2. **Superpowers plans** in `docs/plans/` — detailed implementation plans with Status headers, version-controlled project artifacts

The standup and plan-status skills need to be aware of both sources. The plan MCP tool stays for portability, but the intelligence about where to find plans lives in the skills layer.

## Architecture Principle

**Dumb storage + smart skills.** The MCP server stays as-is. All new intelligence goes into the skills layer. Agents use their native file-reading ability for `docs/plans/` — no new MCP tools needed.

## Change 1: New `/plan` Skill

**Location:** `skills/plan/SKILL.md`

**Purpose:** Guide agents (especially non-Claude Code clients) through using the plan MCP tool effectively.

**Scope:**
- When to create a plan (starting multi-session work, strategic direction for a project)
- How to structure plan content (title, markdown body, tags)
- When to update status (mark completed when done, archive when superseded)
- The active plan concept (one per workspace, shows in recall)
- Explicit guidance to update status when work is complete

**Tone:** Same aggressive behavioral language as other Goldfish skills.

**Allowed tools:** `mcp__goldfish__plan`

---

## Change 2: Update `/standup` Skill — Dual-Source Plan Awareness

**Current behavior:** Calls `recall({ workspace: "all", days: 1 })`, synthesizes checkpoints. Gets active Goldfish plans via recall output. No awareness of `docs/plans/`.

**New behavior:** After recall, the agent also:
1. Checks `docs/plans/*.md` in the current project for plan files
2. Reads the Status field from plan headers
3. Cross-references plan tasks against recent checkpoints to assess actual completion
4. Includes both Goldfish plans (from recall) AND `docs/plans/` plans in the forward-looking section

**Forward-looking section added to report format:**

```
### Up Next
- Lab Handbook: validation logic remaining (from docs/plans/2026-02-14-lab-handbook.md)
- Goldfish: workspace env var implementation (from active Goldfish plan)
```

**Status inference rule:** If a plan's Status header says "Approved" but checkpoints show all tasks completed, note it as effectively complete. If Status says "Approved" and there's no checkpoint activity, it's genuinely upcoming work.

**No MCP tool changes** — the agent reads `docs/plans/` with its native file-reading capability.

---

## Change 3: Update `/plan-status` Skill — Same Dual-Source Awareness

**Current behavior:** Gets the active Goldfish plan via `plan({ action: "get" })`, recalls checkpoints, cross-references to assess progress. Only knows about `.memories/plans/`.

**New behavior:** Same assessment methodology, but also:
1. Scans `docs/plans/*.md` for plan files in the current project
2. Includes those plans in the assessment alongside the active Goldfish plan
3. Uses the Status header as a starting point but verifies against checkpoint evidence
4. Reports on both sources clearly so the user knows which plans live where

The assessment logic (map checkpoints to plan items, categorize as completed/in-progress/not-started, flag drift) stays the same. Just expanding the input surface.

## Change 4: `GOLDFISH_WORKSPACE` Environment Variable

**Purpose:** VS Code / GitHub Copilot users need explicit workspace resolution since `process.cwd()` doesn't reliably point to the project root.

**Implementation:**

1. Add `resolveWorkspace(explicit?: string): string` to `src/workspace.ts`
   - Priority: explicit arg > `GOLDFISH_WORKSPACE` env var > `process.cwd()`
2. Replace all 7 `process.cwd()` fallback sites with `resolveWorkspace()` calls
3. Tests for priority order (explicit > env > cwd, `"current"` treated as undefined)
4. Document VS Code `.vscode/mcp.json` setup in README

**Files touched:**
- `src/workspace.ts` — add `resolveWorkspace()`
- `src/handlers/checkpoint.ts`, `src/handlers/plan.ts`, `src/handlers/recall.ts`
- `src/checkpoints.ts`, `src/plans.ts`, `src/recall.ts`
- `tests/workspace.test.ts` — new tests
- `README.md` — VS Code setup section

**TDD approach:** Write resolver tests first, implement, then replace callsites one at a time.

## What's NOT Changing

- **Checkpoint tool + skill** — working well, no changes
- **Recall tool + skill** — working well, no changes
- **Plan MCP tool** — stays as-is for portability
- **Hooks** — SessionStart, PreCompact, PostToolUse(ExitPlanMode) all stay
- **No task management** — Claude Code handles this natively
- **No DevOps integration** — work items are too high-level to be useful
- **No Anamnesis concerns** — separate project (~/Source/claude), different scope

## Implementation Order

1. `GOLDFISH_WORKSPACE` env var (standalone, unblocks VS Code users)
2. New `/plan` skill (standalone, no dependencies)
3. Update `/standup` skill (depends on understanding plan format)
4. Update `/plan-status` skill (same pattern as standup update)

## Success Criteria

- VS Code users can configure `GOLDFISH_WORKSPACE` and get correct `.memories/` placement
- `/plan` skill guides agents through plan lifecycle including status updates
- `/standup` surfaces forward-looking items from both `.memories/plans/` and `docs/plans/`
- `/plan-status` assesses progress across both plan sources
- All existing 260 tests still pass + new workspace resolver tests
