## Task 3: Auto-attach planId during saveCheckpoint

### What
Added automatic plan affinity to `saveCheckpoint()` -- when an active plan exists, its ID is attached to the checkpoint as `planId`.

### Why
Part of plan-checkpoint affinity feature. Enables checkpoints to be associated with the plan that was active at the time they were created, without requiring the caller to explicitly pass the plan ID.

### How
- Added `import { getActivePlan } from './plans.js'` to `src/checkpoints.ts`
- After the auto-generate summary block and before file path determination, added a try/catch block that calls `getActivePlan(projectPath)` and sets `checkpoint.planId` if a plan is active
- Wrapped in try/catch because plan affinity is best-effort (should not break checkpoint save)

### Tests (2 new)
1. `attaches planId when active plan exists` -- creates a plan with `activate: true`, saves a checkpoint, verifies `planId` matches
2. `omits planId when no active plan exists` -- saves a checkpoint without any plan, verifies `planId` is undefined

- 273 tests passing (all green, up from 271)

### Impact
Checkpoints now automatically carry plan context. Combined with Tasks 1 (type) and 2 (serialization), checkpoints are fully plan-aware end-to-end.
