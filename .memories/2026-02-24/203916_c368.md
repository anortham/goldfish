## Task 4: Add planId filter to recall

### What
Added `planId` optional field to `RecallOptions` and implemented filtering in `recallFromWorkspace()` so recalled checkpoints can be scoped to a specific plan.

### Why
Part of plan-checkpoint affinity feature. Users need to be able to recall only the checkpoints associated with a specific plan, enabling focused context retrieval.

### How
- Added `planId?: string` to `RecallOptions` in `src/types.ts`
- Added a simple filter block in `recallFromWorkspace()` in `src/recall.ts`, placed after fuzzy search (preserves search relevance ordering) but before summary stripping (avoids unnecessary work on filtered-out checkpoints)
- Two tests: positive case (filter returns only matching checkpoint) and negative case (no filter returns all)

### Tests (2 new)
1. `filters checkpoints by planId` -- creates 3 checkpoints with different/no planIds, verifies only matching one returned
2. `returns all checkpoints when planId not specified` -- verifies no regression when planId is omitted

### Result
- 275 tests passing (all green, up from 273)
- Commit: `ab6f494` on `feat/plan-checkpoint-affinity`

### Impact
Recall is now plan-aware. Combined with Tasks 1-3, the full pipeline works: checkpoints get tagged with planId on save, persisted in frontmatter, and can be filtered on recall.
