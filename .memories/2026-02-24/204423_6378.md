## Task 6: Display planId in handler output

### What
Added `Plan: {planId}` line to both checkpoint and recall handler markdown output when a checkpoint has an associated plan.

### Why
Part of plan-checkpoint affinity feature. Without this, the planId was stored on checkpoints but invisible in handler responses -- agents couldn't see which plan a checkpoint belonged to.

### How
- **`src/handlers/recall.ts`**: Added `if (checkpoint.planId)` block in `formatCheckpoint()` helper, after tags line
- **`src/handlers/checkpoint.ts`**: Added `if (checkpoint.planId)` block in `handleCheckpoint()`, after tags line
- **`tests/handlers.test.ts`**: Added 2 tests (one per handler) that create an active plan, save a checkpoint, and verify the output contains `Plan: {plan-id}`

### Tests
1. `shows planId in checkpoint output when present` (recall handler)
2. `shows planId in checkpoint response when active plan exists` (checkpoint handler)

### Result
- 277 tests passing, 0 failures
- Commit: `092ef0c` on `feat/plan-checkpoint-affinity`

### Impact
Agents now see plan context when checkpointing or recalling. This completes the visibility loop for plan-checkpoint affinity.
