## Addressed all 4 review findings on recall last-N mode

### What was done
Fixed all 4 issues identified in the code review of the recall last-N mode changes:

1. **`getAllCheckpoints()` early termination** — Added optional `limit` parameter. Scans newest directories first, stops once enough checkpoints accumulated, slices to exact limit after sorting.

2. **Dead `|| 2` fallback** — Removed misleading `options.days || 2` default in `getDateRange()`. Since `hasDateParams()` gates the call, `options.days` is always set when this code runs. Changed to `options.days!`.

3. **Registry concurrent test ENOENT on Windows** — Root cause: rapid `writeFile → rename` to the same destination file causes Windows ENOENT errors. Fix: replaced atomic write-then-rename with direct `writeFile` in both `registerProject` and `unregisterProject`. Safe because the lock provides exclusivity, and `getRegistry` already handles corrupted files gracefully.

4. **Cross-workspace recall loading everything** — Changed per-project limit from `99999` to actual global limit. Each project returns at most `limit` checkpoints (newest first), then merged results are sliced to global limit. Also passes `earlyLimit` through to `getAllCheckpoints` (skipped when searching, since fuse.js needs all candidates).

### Impact
- **253 tests passing, 0 failures** (was 248 pass / 1 fail before)
- 4 new tests for `getAllCheckpoints` (limit, overshoot, empty project)
- Performance improvement for projects with many months of checkpoints
- Registry concurrent test no longer flaky on Windows
