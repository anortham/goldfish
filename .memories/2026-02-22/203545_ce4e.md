## Added new high-priority finding: checkpoint over-frequency with OpenCode + GPT

Captured user-reported UX issue that checkpoint is being called too frequently in real usage.

- **What:** Added a new review item noting `checkpoint()` cadence is likely too aggressive (especially with OpenCode + GPT).
- **Why:** Current language emphasizes checkpointing after *every* task and many conversational moments, which can create noise and reduce signal quality in memory history.
- **How this manifests:**
  - Excessive `.memories/` churn
  - Lower-value checkpoints crowding out meaningful milestones
  - Potential user friction from repetitive tool calls
- **Proposed direction:** Reframe guidance toward **high-value boundaries** (decision points, completed deliverables, blockers, risky transitions, pre-compaction) and de-emphasize micro-step checkpointing.
- **Impact:** Should improve memory signal-to-noise, reduce operational noise, and preserve continuity quality without losing critical context.
