## Updated checkpoint guidance to milestone-focused language

Implemented wording changes to reduce checkpoint over-frequency and emphasize higher-value continuity events.

- **What changed:**
  - Rewrote checkpoint tool description in `src/tools.ts` to prioritize milestone-level checkpoints, key decisions, blockers, and crash/context-loss resumability.
  - Updated server behavioral instructions in `src/instructions.ts` from "checkpoint after every task" to milestone/decision/continuation boundaries.
  - Revised `skills/checkpoint/SKILL.md` to align with the same positive framing and continuity-quality focus.
  - Updated `tests/server.test.ts` assertions to validate the new milestone-oriented wording.
- **Why:** Frequent low-value checkpoints were creating noise in real OpenCode + GPT workflows; the goal is better signal-to-noise without losing continuity.
- **How:** Kept strong mandatory posture while shifting trigger criteria to meaningful states instead of micro-steps.
- **Impact:** Guidance now encourages fewer, higher-value checkpoints that are more useful for session recovery and cross-session continuation. Verified with full test suite (`bun test`, all passing).
