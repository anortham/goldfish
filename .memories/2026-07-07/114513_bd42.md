## Fixed lock mutual-exclusion race + Windows rename retries; released as 7.4.3

While preparing the 7.4.3 release (originally just the Bun 1.3 `prepare` script fix), the full suite surfaced 11 pre-existing failures on Windows under Bun 1.3. Root-caused all of them; two were real product bugs.

### Product fixes
- **Lock steal race (`src/lock.ts`):** `writeFile(lockPath, data, {flag:'wx'})` creates the lock file before content lands. A concurrent waiter reading in that window got `''`, JSON.parse failed, the lock was judged malformed/stale, and a LIVE lock was stolen — two holders in the critical section. Measured 45/500 concurrent reads seeing the empty window; caused 4-of-5 lost registry updates. Fix: malformed locks get a 1s mtime grace period (presumed mid-write) before they can be stolen; old malformed locks are still reclaimed promptly.
- **Atomic write rename (`src/file-io.ts`):** Windows rename-over-existing fails EPERM/EACCES/EBUSY transiently when the destination is briefly held open (concurrent replace, unclosed reader, AV scan). Fix: bounded retry (10 × 5ms) on win32 only, plus temp-file cleanup when the rename ultimately fails.

### Test fixes (platform-dependence exposed by Bun 1.3 on Windows)
- Bun 1.2 returned forward-slash paths from mkdtemp/tmpdir on Windows and accepted drive-less `file:///` URLs in fileURLToPath; Bun 1.3 returns native backslashes and rejects drive-less URLs (matching Node). Tests that relied on the old behavior were fixed: separator-insensitive path assertions (server, recall, workspace), `resolve()`-based roots fixtures, a rewritten registry drive-case dedup test using real drive-letter case variants, and win32 skips for the two chmod-based lock tests (directory write bits aren't enforced on Windows).

### Impact
Concurrent checkpoint/registry writes on Bun 1.3/Windows no longer lose data. Full suite: 574 pass / 0 fail / 2 POSIX-only skips; typecheck clean. Released as 7.4.3.
