## Fixed workspace recovery review findings

### WHAT
- Added regression coverage for cwd recovery when an outer registered project contains an unregistered nested repo with its own `.git` marker.
- Added regression coverage for Windows-style path fallback matching when `realpath` fails and drive-letter casing differs.
- Updated recovery to compare registry and parent-walk candidates and choose the deeper project evidence, while keeping `registry` when both candidates identify the same path.
- Updated parent walking to use `win32` path operations for Windows-looking paths so POSIX test runs do not accidentally walk to `.`.
- Updated the design note and changelog to describe the nearest-evidence rule.

### WHY
The review found two path-hardening gaps: registry-first ordering could silently write to an outer project instead of a nested repo, and fallback path comparison missed Windows drive-case parity.

### HOW
- Introduced `normalizePathKeyForSafetyCheck` from the existing unsafe-cwd normalization rules.
- Used realpath-or-normalized path keys in `workspace-recovery.ts` ancestor matching and candidate specificity checks.
- Preserved the caller-facing behavior: explicit workspace, `GOLDFISH_WORKSPACE`, and MCP roots still win before cwd recovery; single registered project fallback remains recall-only.

### IMPACT
The cwd/roots fix is now safer for Cursor/plugin-style launches and nested workspaces: Goldfish recovers a nearby real project root without allowing an outer registered repo to steal writes.
