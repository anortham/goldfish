## Brief Repositioning Implementation Plan

### WHAT
Added `docs/plans/2026-04-16-brief-repositioning-implementation.md`, a task-by-task implementation plan for migrating Goldfish from `plan` to `brief`.

### WHY
The approved design needs a concrete execution path across storage, MCP tools, recall/checkpoint behavior, hooks, skills, docs, and release metadata. Without that, the rename stays a design memo and never lands.

### HOW
- Chose a light implementation plan with six tasks: storage/types, MCP tool surface, recall/checkpoint/consolidation semantics, hooks/skills, docs/release metadata, and verification
- Kept the migration compatibility-first: new public `brief` surface, legacy `plan` aliases for one release, lazy storage migration, and legacy `planId` reads
- Locked in exact file targets and verification commands so execution can proceed without re-scoping the work

### IMPACT
- Turns the product decision into an executable sequence instead of vague intent
- Reduces migration risk by making compatibility an explicit deliverable
- Sets up either same-session execution or a clean handoff to a separate execution session
