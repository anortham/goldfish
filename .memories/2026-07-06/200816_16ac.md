## Added timeout for stuck MCP roots/list lookup

### WHAT
Bound request-time MCP `roots/list` hydration with an internal timeout and added a regression for a roots request that never settles. Also fixed the package build script to target Bun instead of Node, because the server and source use Bun runtime APIs.

### WHY
A dead or desynced MCP client connection can leave `sendRequest({ method: 'roots/list' })` pending forever. Since workspace-aware tools await roots before falling back to cwd/recovery, `/recall`, `checkpoint`, or `brief` could hang without ever reaching the normal recovery path.

### HOW
- Added `ROOTS_LIST_TIMEOUT_MS = 500` in `src/server.ts`.
- Wrapped `sendRequest({ method: 'roots/list' }, ListRootsResultSchema)` in `Promise.race`.
- Timed-out roots lookup returns `undefined`, preserving existing fallback and retry behavior because empty/failed roots are not cached.
- Cleared the timeout handle in `finally` so fast success/failure does not leave timers behind.
- Added a server regression test with a `roots/list` handler that returns a never-settling promise and confirms checkpoint falls back to cwd.
- Corrected `package.json` build target from `node` to `bun` after `bun run build` exposed the existing mismatch with Bun builtins.

### IMPACT
Workspace-aware tool calls no longer hang indefinitely just because the client-side roots request never answers. Normal roots behavior, roots retries, cwd fallback, and workspace recovery remain covered by the existing server suite.
