## Verified Windows registry duplicate fix

### WHAT
Reviewed the user-provided broken `registry.json` backup and diagnosis, then checked the committed duplicate-Windows-entry fix in `src/registry.ts` against live code paths and tests.

### WHY
The reported registry had duplicate workspace entries split by Windows drive-letter casing and path separators. We needed to confirm the applied fix handles the real messy-registry shape, not only a POSIX-degenerate test case.

### HOW
- Parsed the supplied backup and confirmed 29 entries, 10 lowercase `c:` paths, one backslash path, and four canonical duplicate groups: MyraNext, Collections, RASDNuGetPackages, Documentation.
- Verified `registryPathKey()` delegates to `normalizePathKeyForSafetyCheck()`, which lowercases Windows paths but preserves POSIX case.
- Verified `registerProject`, `unregisterProject`, and `listRegisteredProjects` use the normalized key, and that recall/server hydration call `listRegisteredProjects`.
- Added a regression test that seeds literal `C:/...` and `c:/...` registry entries and creates both matching fixture directories so the test fails with the old case-sensitive comparison even on POSIX.
- Red check: temporarily restored case-sensitive key behavior and the new test failed with received length 2.
- Green check: restored the fix and the new test passed.

### IMPACT
The registry fix is real for duplicate workspace registration and cross-workspace recall double-scanning. It does not, by itself, fix a dead/desynced MCP client connection to an exited server; `getCachedRoots()` still awaits `roots/list` without a timeout and remains a separate hardening follow-up.
