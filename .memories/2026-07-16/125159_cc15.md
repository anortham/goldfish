## Track 3: performance + token fixes — all CLAUDE.md targets met

Third review-fix track (plan: docs/plans/2026-07-16-review-fixes-plan.md). Post-fix benchmark on a synthetic 1k-checkpoint corpus vs the review's baseline measurements:

- **Search: 247ms → 11ms warm** (139ms first-ever build, 73ms after a corpus change). Two layers: a per-day corpus cache in checkpoints.ts (fingerprint = name+size+mtime of every file, validated on each read — external edits always picked up, nothing derived on disk) and an Orama index cache in ranking.ts keyed by workspace + corpus fingerprint, only for whole-corpus searches (filtered subsets always build fresh; cached hits are mapped through the passed set so extra docs can never leak).
- **Structured filters: 122ms → ~10ms** via the same day cache.
- **Checkpoint save: 43ms → ~21ms and non-blocking** — getGitContext is now async Bun.spawn with all 5 git queries in parallel (was 5 serial spawnSync calls blocking the event loop for ~40ms).
- **limit:0 cliff removed:** brief-activity scan bounded to 28 days (STALE_SCAN_MAX_DAYS) — deep-history activity can't change the stale verdict; display falls back to brief.updated.
- **Token caps:** full-mode recall response budgeted at 20k chars with an explicit "(output truncated: showing N of M)" note (was uncapped ~48k chars at limit:20); default-mode `next` truncated at 140 chars.
- **Digest dedup:** buildCompactSearchDescription composes atomic parts with containment-aware dedup — headings that prefix the decision no longer burn ~27% of the 220-char budget.
- **Mechanical:** parallel file reads in day loading, parallel registry stats, registerProject skips the global lock when already registered.

TDD: watched failing tests first for index-cache stats, day-cache freshness (in-place edit), bounded stale scan (99d vs 40d display), full-mode cap, next truncation, digest dedup.
